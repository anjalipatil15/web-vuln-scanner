The OWASP Top 10

The OWASP Top 10 is the industry's standing list of the most critical risks to web applications, rebuilt every few years from real vulnerability data and practitioner surveys. This page covers the current 2025 edition, published in January 2026 — the first refresh since 2021.

Full detail lives at owasp.org/Top10/2025.

01

A01:2025 — Broken Access Control

The app fails to properly enforce what a signed-in user is allowed to see or do, so people can reach data or actions meant for someone else — often just by editing an ID in a URL. Server-side request forgery, previously its own category, was folded in here for 2025.

Looks like: viewing another user's record by changing an ID in the address bar · an admin action that trusts a hidden form field instead of a server-side role check · a server that will fetch any URL it's given, including internal ones.

02

A02:2025 — Security Misconfiguration

Security controls exist but are set up wrong, left at insecure defaults, or simply missing. This jumped from #5 to #2 in 2025 as more application behavior has come to depend on configuration rather than code. This scanner's header and cookie checks fall under this category.

Looks like: a missing Content-Security-Policy header · a session cookie issued without the Secure or HttpOnly flag · a stack trace leaking to visitors on error pages.

03

A03:2025 — Software Supply Chain Failures

Risk introduced through the chain of dependencies, build tools, and package registries an app relies on — a broader successor to 2021's "Vulnerable and Outdated Components," covering the whole ecosystem rather than just unpatched libraries.

Looks like: a dependency with a known, unpatched CVE · a CI pipeline pulling unpinned packages at build time · a compromised package slipped into a lockfile.

04

A04:2025 — Cryptographic Failures

Sensitive data ends up exposed because it was never encrypted, was protected with a broken or outdated algorithm, or had its keys handled carelessly.

Looks like: passwords hashed with a fast, unsalted algorithm · card numbers stored in plaintext · a server still willing to negotiate old, broken TLS versions.

05

A05:2025 — Injection

Untrusted input reaches an interpreter, query, or page without being validated or escaped first, letting an attacker change what the underlying system executes. This spans everything from Cross-Site Scripting to SQL injection — the two attack classes this scanner actively tests for.

Looks like: a search box that echoes input straight back into the page · a login field vulnerable to a crafted SQL string · a URL parameter passed unescaped into a database query.

06

A06:2025 — Insecure Design

The vulnerability was built into the architecture before any code was written — usually a missing threat model or an assumption about user behavior that never held up in practice.

Looks like: a password-reset flow with no rate limiting · a checkout that trusts the client to report the order total · no plan for isolating one customer's data from another's in a multi-tenant system.

07

A07:2025 — Authentication Failures

Weaknesses in how an application confirms who's making a request — renamed from 2021's "Identification and Authentication Failures" to better match what the category actually covers.

Looks like: no lockout after repeated failed logins · session tokens that don't rotate after sign-in · predictable password-reset tokens.

08

A08:2025 — Software or Data Integrity Failures

Code or data gets trusted without verifying it hasn't been tampered with — a lower-level counterpart to Software Supply Chain Failures, focused on integrity checks at the individual artifact level.

Looks like: an auto-updater that installs packages without checking a signature · a CI/CD pipeline with no protection against unauthorized changes · deserializing untrusted data straight into application objects.

09

A09:2025 — Security Logging & Alerting Failures

Suspicious activity happens but is never logged — or it's logged with nobody and nothing watching. Renamed in 2025 to make clear that logging without alerting isn't enough on its own.

Looks like: failed logins that leave no audit trail · logs that exist but are never reviewed · a breach that runs for months before anyone notices.

10

A10:2025 — Mishandling of Exceptional Conditions

New for 2025. Applications behave unpredictably or insecurely when something goes wrong — errors, unexpected input, or edge cases nobody planned for.

Looks like: a raw stack trace returned to the browser on a server error · a payment step that fails "open" and lets the transaction through anyway · an API that crashes instead of cleanly rejecting a malformed request.
